1 Description

You’ve inherited a garage full of vintage video games (cartridges, floppy disks in huge boxes, jewel cased CD-ROM). Rather then letting them decay, you have decided to dedicate all your free time to archiving them digitally by building a interactive web site that lets users view, search, add, and manage a catalogue of classic games.

Your task is to build a single page web application that will allow users to view and manage the catalogue. You will need to use the following technologies:

  • HTML5 for the structure of web pages,

  • CSS3 for presentation of web pages,

  • JavaScript for client-side programming.

This task is split into several parts of increasing complexity. You are strongly encouraged to complete them in the stated order and not move onto the more advanced tasks until you are happy with your solution to the more basic tasks. The tasks build on each other and should be submitted as a single project.

You are not allowed to use 3rd party libraries, frameworks, or preprocessors (such as React or SCSS).

The suggested structure is that there is an index.html, a style.css, and one or more JavaScript files. However, you should feel free to diverge from this as you see fit and can justify in your report.

Your application must be runnable via a locally running webserver on the lab machines. You can serve a local webpage on a lab machine by running the following command, "python3 -m http.server port number" in the folder that you want to serve.

2 Data Model

Some of the games have already been catalogued and their information stored as JavaScript object literals, one for each game. This current state of the catalogue is provided in the file games.js (found on Studres).

Each object stores the following information:

  • Title: the name of the game, a string that must be less than or equal to 40 characters in length.

  • Platform: The type of computer or console the game runs on. This can have one of the following values: NES, SNES, PS1, PC, or AMIGA.

  • Genre: the genre the game is in, a string that must be less than or equal to 20 characters in length.

  • Description: A short description of the game, a string that must be less than or equal to 200 characters in length.

Note that all of these fields are required and cannot be missing or left blank.

You are free to load and transform games.js as you see fit, selecting appropriate data structures for your implementation. However, entries in the catalogue must conform to the above constraints for their properties. Do not edit games.js, though you may include it in your project. If you do, please ensure that it is included in your submission. Note that this application is not required to persist the data between reloads of the page.

3 Requirements

3.1 Basic Functionality

Create a web application using HTML, CSS, and client-side JavaScript and do the following:

  • Add a list containing all the games from the games.js file displaying the title, platform, title, and description. When doing this you must ensure that all the games in the list conform to the constraints stated in the data model. If they do not, you should log an error to the console, not display the entry, and then continue processing the other entries. Note that this list is not required to be an HTML list, however you should justify your selection of an appropriate element type in your report.

  • Design and implement a way for the user to add new games to the list.

  • Design and implement a way for the user to remove games from the list.

  • In a seperate CSS file, add CSS styles to the list entries to make them more visual appealing based on your own tastes. Make sure to set the font, background, and text color at a minimum.

3.2 Standard Functionality

  • Validation: Ensure that any new entry that is added conforms to the constraints stated in the data model. If it does not then the action should be disallowed and user notified.

  • Duplicate Detection: Prevent the user from adding a game that already exists in teh catalogue. Two games are considered the same if they have the same values for their title and platform fields.

  • Sorting: Design and implement a way for the user to sort the list by name, platform, or genre.

3.3 Advanced Functionality

  • Edit: Design and implement a way for the user to modify the properties of an existing game. Any new values should be validated (as in part 2) to ensure that they conform to the data model.

  • Search: Design and implement a way for the user to search the catalogue for a specific item or set of items based on the name. Partial or case-insensitive queries and how search interacts with sorting should be handled gracefully and how this is done should be justifed in your report.

4 Report

Your report must show, evidence of testing, including descriptions of any issues you encountered and how they were resolved.

Your report must use the structure outlined in the report_template directory on studres. In particular you should pay attention to how to structure your testing section.

5 Hints

We recommend an incremental approach: make sure that each part of your program works before moving on to the next part. Make a safe copy of whatever you have got working so far so that you always have a working solution to submit while you are developing the next part of your solution. This should allow you to experiment with different ways of organising your code and implementing functionality without worrying of breaking things.

  • Use an HTML validator and a CSS validator to validate your code.

  • Use strict mode for JavaScript and check for errors/warnings in the JavaScript console of your web browser.

  • It is good practice to write your HTML code, CSS code, and JavaScript code in separate files and link to the CSS and JavaScript files from within the HTML file as appropriate.

  • Use the Live Server plugin for Visual Studio Code (or something similar for your editor or IDE of choice) to run your web application when developing it.

  • Test using the local web server (as described in the description) on the lab machines.

  • Any requirement requiring CSS is designed to assess your ability to code using CSS rather than your graphic or interactive design skills. You should aim for a clean, simple, and consistent design but we will be grading the CSS based on the quality of the code rather than the visual appearance.

Submission

A single file containing your code and PDF report in ZIP format must be submitted electronically via MMS by the deadline. Submissions in any other format will be rejected.

The specific mark descriptors for this piece of coursework are described below.

  • A poor implementation in the 7-10 grade band will contain part of the solution that demonstrates some understanding of main concepts but does not work reliably and misses much of the functionality needed for Basic Functionality.

  • A basic implementation in the 11–13 grade band will complete the Basic Functionality. The report should demonstrate understanding of basic concepts. The lower parts of this mark range will correspond to submissions where there are weaknesses or the report demonstrates a lack of understanding.

  • An implementation in the 14–16 range should complete the Basic and Standard Functionality, including a clear description of what is done and why. A mark of 16 will correctly implement all requirements for both parts successfully, with understandable code and an insightful report with high-quality documented testing.

  • An implementation in the 17–18 will complete all requirements. This grade band requires an excellent implementation of the Basic and Standard Functionality and a solid, non-trivial progress on the Advanced Functionality including a report that demonstrates understanding of those concepts with high-quality documented testing.

  • A grade of 19 and higher will require completion of all parts and evidence deep understanding of the design, implementation, and testing of the application detailed in the report.
